# 照片处理 GUI 工具方案文档

## 1. 文档目标

本文档用于说明项目首版的整体解决方案，回答“为什么这样做”和“系统准备怎么做”。

本方案以当前需求文档为基线，目标是在不过度设计的前提下，快速交付一个可用、可扩展、风险可控的本地桌面工具。

## 2. 方案概述

项目采用 Python + PyQt 构建本地 GUI 应用，聚焦两个高频场景：

1. 批量修改照片时间信息
2. 根据成片目录保留原片并清理未匹配文件

方案重点不在复杂 UI，而在以下四点：

- 预览优先
- 批处理稳定
- 删除安全
- 双平台可运行

## 3. 方案原则

### 3.1 本地优先

- 所有文件处理均在本地完成
- 不依赖网络服务
- 不上传照片元数据或文件内容

### 3.2 预览优先

- 所有会修改文件状态的操作都先计算预览结果
- 执行前用户必须明确确认
- 预览和执行使用同一任务模型，避免结果漂移

### 3.3 安全优先

- 删除操作默认推荐移动到回收站
- 永久删除保留为显式可选模式
- 单文件失败不影响整个任务完成

### 3.4 简单优先

- 首版不做插件化
- 首版不做数据库
- 首版不做多进程分布式处理
- 首版不做复杂工作流编排

## 4. 核心方案

### 4.1 时间修改方案

采用“扫描 -> 计算偏移 -> 生成预览 -> 用户确认 -> 写回文件”的方案。

关键点：

- 把创建时间、修改时间、拍摄时间视为三类独立字段
- 每个字段支持独立勾选
- 偏移量以统一模型计算，再映射到具体时间字段
- 对无法写入的字段单独报错，不中断其他文件

该方案的好处是：

- 规则清晰
- 预览结果可解释
- 易于定位失败原因

### 4.2 原片保留方案

采用“扫描成片 -> 生成 basename 集合 -> 扫描原片 -> 分类预览 -> 用户确认 -> 执行删除/回收站移动”的方案。

关键点：

- 匹配规则固定为 basename 匹配
- 忽略扩展名
- 原片目录中同 basename 的所有文件都视为应保留对象
- 删除策略作为独立执行模式切换

该方案的好处是：

- 规则稳定，不依赖具体相机品牌或扩展名枚举
- 易于向用户解释保留逻辑
- 后续扩展 sidecar 规则时也有清晰入口

## 5. 技术方案

### 5.1 桌面框架

- GUI：PyQt6
- 语言：Python 3.12
- 运行方式：必须在项目 `.venv` 中运行

选择理由：

- PyQt 对桌面表格、文件选择、进度反馈、线程协作支持成熟
- Python 生态便于处理 EXIF、文件系统和平台差异

### 5.2 任务执行方式

使用 GUI 主线程 + 后台工作线程的模式：

- 主线程负责界面交互与状态展示
- 后台线程负责扫描、预览计算、执行写入或删除

这样可以避免大量文件处理时界面卡死。

### 5.3 文件处理策略

- 扫描阶段只读取必要信息
- 预览阶段只做计算，不修改文件
- 执行阶段按预览结果逐项落地
- 每个文件独立记录结果

### 5.4 日志与结果展示

- GUI 中展示摘要结果
- 保留失败项明细
- 后续可扩展导出执行结果，但首版不强制

## 6. 风险与应对

### 6.1 文件时间跨平台差异

风险：

- macOS 与 Windows 对创建时间写入机制不同

应对：

- 在文件时间写入层做平台隔离
- 文档中明确失败可见，不允许静默失败

### 6.2 EXIF 写入差异

风险：

- 并非所有格式都支持 EXIF 读写

应对：

- 预览阶段提前识别
- 执行阶段逐文件记录失败原因

### 6.3 删除误操作风险

风险：

- 用户可能误删原片

应对：

- 默认推荐回收站模式
- 必须预览
- 必须二次确认
- 结果页显示完整统计

### 6.4 Python 环境不一致

风险：

- 使用系统 Python 可能导致依赖版本不一致

应对：

- 所有开发与运行流程统一要求在项目 `.venv` 中执行
- 后续脚本、README、启动流程统一围绕 `.venv` 编写

## 7. 交付方案

首版建议按以下顺序推进：

1. 建立项目骨架与 `.venv` 约束
2. 建立基础 GUI 框架
3. 实现时间修改的扫描与预览
4. 实现时间修改执行
5. 实现原片保留的扫描与预览
6. 实现原片保留执行
7. 完成异常处理、结果页与测试

## 8. 结论

该方案以“少而稳”为原则，优先解决两个高频场景，并通过统一任务模型、预览先行和安全删除策略控制风险。后续架构与实施计划应围绕这一方案展开，不额外扩展产品边界。
