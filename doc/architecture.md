# 照片处理 GUI 工具架构文档

## 1. 架构目标

本项目架构需要满足以下目标：

- 支持两个独立功能模块并共享基础能力
- 保证 GUI 不被批处理任务阻塞
- 保证预览与执行逻辑一致
- 便于后续扩展更多文件处理任务
- 强制项目运行在 `.venv` 虚拟环境中

## 2. 总体架构

建议采用分层结构：

1. GUI 层
2. 应用服务层
3. 领域任务层
4. 基础设施层

## 3. 分层职责

### 3.1 GUI 层

负责：

- 主窗口
- 两个功能页
- 文件夹/文件选择
- 参数输入
- 预览表格
- 确认对话框
- 进度显示
- 结果展示

GUI 层不直接处理文件写入、删除或 EXIF 修改，只通过服务层发起任务。

### 3.2 应用服务层

负责：

- 接收 GUI 输入
- 组织任务参数
- 触发预览任务
- 触发执行任务
- 汇总结果并回传 GUI

这一层是 GUI 与底层处理逻辑之间的协调层。

### 3.3 领域任务层

负责实现核心业务规则：

- 时间修改任务
- 原片保留任务
- 预览记录生成
- 结果统计
- 错误分类

该层不依赖具体 GUI 组件，保持可测试性。

### 3.4 基础设施层

负责封装底层能力：

- 文件扫描
- 文件时间读取/写入
- EXIF 读取/写入
- 回收站删除
- 永久删除
- 平台差异适配
- 虚拟环境检查

## 4. 推荐目录结构

建议初始目录结构如下：

```text
photo-tools/
  AGENTS.md
  doc/
  src/
    app/
    ui/
    services/
    tasks/
    infrastructure/
    models/
  tests/
  requirements.txt
  README.md
  .gitignore
```

说明：

- `src/ui`：GUI 页面和窗口
- `src/services`：应用服务与任务编排
- `src/tasks`：业务任务核心逻辑
- `src/infrastructure`：文件系统、EXIF、平台适配
- `src/models`：请求、预览、结果等数据模型

## 5. 核心模块设计

### 5.1 时间修改模块

建议拆为以下职责：

- 扫描器：收集目标文件与时间元数据
- 预览生成器：根据偏移量生成前后对比
- 执行器：按勾选字段写回时间
- 结果汇总器：汇总成功、失败、跳过

### 5.2 原片保留模块

建议拆为以下职责：

- 成片扫描器：提取 basename 集合
- 原片扫描器：列出可处理文件
- 匹配器：判断保留或待处理
- 执行器：按策略移动到回收站或永久删除
- 结果汇总器：统计执行结果

### 5.3 通用模块

建议抽取以下通用能力：

- 任务进度通知
- 错误对象与错误码
- 预览记录模型
- 任务结果模型
- 平台能力探测

## 6. 数据模型建议

### 6.1 时间修改请求

字段建议：

- `files`
- `offset_days`
- `offset_hours`
- `offset_minutes`
- `offset_seconds`
- `update_created_at`
- `update_modified_at`
- `update_taken_at`

### 6.2 时间修改预览记录

字段建议：

- `path`
- `name`
- `before_created_at`
- `after_created_at`
- `before_modified_at`
- `after_modified_at`
- `before_taken_at`
- `after_taken_at`
- `status`
- `message`

### 6.3 原片保留请求

字段建议：

- `selected_dir`
- `source_dir`
- `delete_mode`

### 6.4 原片保留预览记录

字段建议：

- `path`
- `name`
- `stem`
- `matched`
- `planned_action`
- `message`

## 7. 并发与线程模型

推荐使用 PyQt 的线程能力处理长任务。

建议原则：

- GUI 主线程只做渲染和交互
- 扫描、预览和执行放入后台线程
- 通过 signal/slot 回传进度、状态和结果

这样可以保证：

- 界面不冻结
- 用户可以看到实时进度
- 错误和结果回传路径清晰

## 8. 环境与运行约束

项目必须统一在 `.venv` 中运行。

建议要求：

- 在项目根目录创建 `.venv`
- 开发命令统一基于 `.venv/bin/python` 或激活后的 `python`
- 启动入口在检测到未进入虚拟环境时给出明确提示
- 测试与工具链同样依赖 `.venv`

## 9. 测试架构

建议测试分层如下：

- 单元测试：任务逻辑、匹配逻辑、偏移计算
- 集成测试：文件扫描、EXIF 读写、删除策略
- GUI 测试：关键流程与状态切换

重点验证：

- 预览与执行一致性
- 单文件失败隔离
- 平台差异下的可见错误
- 高风险操作确认流程

## 10. 架构决策总结

当前架构选择的核心是：

- 用分层结构隔离 GUI 和文件处理逻辑
- 用统一任务模型保证预览与执行一致
- 用基础设施层封装平台差异
- 用 `.venv` 约束统一开发与运行环境

这套架构适合当前空项目启动，也适合后续逐步扩展更多照片处理能力。
