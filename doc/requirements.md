# 照片处理 GUI 工具需求文档

## 1. 文档信息

- 项目名称：照片处理 GUI 工具
- 项目阶段：需求定义
- 目标版本：V1
- 技术框架：Python、PyQt
- Python 运行约束：必须在项目 `.venv` 虚拟环境中运行
- 目标平台：macOS、Windows
- 文档语言：中文

## 2. 项目背景

该项目用于支持个人日常照片整理流程，重点解决两个高频问题：

1. 批量修正照片的时间信息，避免导入、备份或多设备拍摄后出现时间错乱。
2. 根据已筛选出的成片，清理原片目录，只保留与成片对应的源文件，减少存储占用与人工筛选成本。

项目首版定位为本地桌面 GUI 工具，不依赖联网服务，不引入照片管理库，不承担图像编辑功能。

## 3. 产品目标

### 3.1 核心目标

1. 提供可视化、可预览、可确认的批量时间修改能力。
2. 提供基于成片目录的原片保留与清理能力。
3. 所有高风险操作都必须在执行前完成预览与二次确认。
4. 支持 macOS 与 Windows 的本地使用。

### 3.2 非目标

以下内容不在 V1 范围内：

- 图像内容编辑、滤镜、裁剪、调色
- 图片格式转换、压缩、重命名
- 云同步、账号系统、在线备份
- 照片库管理、标签、搜索、相册系统
- 自动识别人脸、场景或画面内容

## 4. 目标用户与使用方式

### 4.1 目标用户

- 有摄影或照片整理习惯的个人用户
- 需要处理本地照片文件夹的用户
- 希望在执行前明确看到改动结果的用户

### 4.2 使用方式

- 单机离线运行
- 用户手动选择本地文件或文件夹
- 工具完成扫描、预览、确认、执行、结果展示

## 5. 功能范围

V1 仅包含以下两个功能模块：

1. 批量修改照片时间信息
2. 根据成片目录保留原片并清理未匹配文件

建议 GUI 主界面以两个页签或两个主操作区承载两个功能模块。

## 6. 功能需求一：批量修改照片时间信息

### 6.1 功能目标

允许用户对一批照片的以下时间信息进行统一偏移调整：

- 文件创建时间
- 文件修改时间
- 拍摄时间

执行前必须提供逐文件预览，让用户确认每个文件修改前后的结果。

### 6.2 输入方式

用户可以通过以下方式选择待处理对象：

- 选择多个文件
- 选择单个文件夹并扫描其中照片

需求文档默认不要求 V1 实现递归扫描子目录，但实现阶段可作为可选增强项评估。

### 6.3 支持的时间字段

V1 需支持以下时间字段的独立勾选：

- 创建时间
- 修改时间
- 拍摄时间

拍摄时间默认指照片 EXIF 中的拍摄时间字段，优先以 `DateTimeOriginal` 作为主要目标字段；若实际实现需兼顾其他关联 EXIF 时间字段，应保持行为一致并在实现说明中写明。

### 6.4 修改策略

V1 默认采用“按偏移量批量平移”的方式。

用户需要输入一个时间偏移量，形式上至少支持：

- 正向偏移，例如 `+2 小时`
- 负向偏移，例如 `-3 分钟`
- 组合偏移，例如 `+1 天 2 小时`

需求层面要求实现支持“天 / 小时 / 分钟 / 秒”维度的偏移输入。GUI 可用表单拆分输入，也可用统一表达式输入，但必须避免歧义。

### 6.5 预览要求

执行前必须生成文件级列表预览，每一行至少展示：

- 文件名
- 文件路径
- 原始创建时间
- 修改后创建时间
- 原始修改时间
- 修改后修改时间
- 原始拍摄时间
- 修改后拍摄时间
- 状态
- 备注

说明：

- 若某个时间字段未被用户勾选，则可显示为“不修改”。
- 若某文件不存在 EXIF 拍摄时间，预览中需明确标记，例如“无 EXIF 时间”。
- 若文件不可写、无法解析、格式不支持，也需在预览中标记状态与原因。

### 6.6 执行要求

用户确认后，系统对选中照片批量执行时间修改。

执行阶段要求：

- 对每个文件独立处理，不因单个文件失败而中断整个任务。
- 支持进度显示。
- 支持结果汇总。
- 支持失败项列表展示。

### 6.7 结果展示

执行完成后，应展示：

- 总文件数
- 成功数量
- 失败数量
- 跳过数量
- 失败文件列表及失败原因

### 6.8 异常与边界

需要覆盖以下情况：

- 文件无 EXIF 拍摄时间
- 文件格式不支持 EXIF 写入
- 文件只读
- 文件被其他程序占用
- 文件路径无权限访问
- 用户输入偏移量为空或非法
- 某些字段可改、某些字段不可改的部分成功场景

### 6.9 验收标准

满足以下条件时视为该功能通过验收：

1. 用户能够选择文件或目录并生成预览。
2. 预览中能看到逐文件的修改前后结果。
3. 执行结果与预览一致。
4. 单文件失败不会导致整个任务停止。
5. 异常文件有明确原因反馈。

## 7. 功能需求二：根据成片目录保留原片并清理未匹配文件

### 7.1 功能目标

根据用户选定的“成片文件夹”，从“原片文件夹”中保留所有与成片同名主体的源文件，并处理其余未匹配文件。

### 7.2 基本定义

- 成片文件夹：用户最终保留或导出的照片所在目录
- 原片文件夹：待清理的源文件目录

### 7.3 匹配规则

V1 默认匹配规则为：

- 仅按文件名主体匹配
- 忽略扩展名

示例：

- 成片目录存在 `IMG_0001.jpg`
- 原片目录中的 `IMG_0001.raw`、`IMG_0001.png`、`IMG_0001.arw` 应全部保留

只要 basename 相同，即视为对应文件。

### 7.4 处理目标

系统需要对原片文件夹中的文件进行分类：

- 保留文件：与成片目录中任一文件名主体匹配的文件
- 待处理文件：未匹配的文件

V1 默认面向照片源文件本体。是否连带保留 sidecar 文件（例如 `.xmp`）在当前需求中不强制要求，后续实现时需在设计文档中明确。如果未特别实现，默认不将 sidecar 作为强制保留对象。

### 7.5 删除策略

V1 同时支持两种处理模式：

1. 移动到回收站
2. 永久删除

默认推荐使用“移动到回收站”。

### 7.6 预览要求

执行前必须生成预览，至少展示：

- 成片目录路径
- 原片目录路径
- 成片识别到的文件数量
- 原片扫描到的文件数量
- 将保留的文件数量
- 将处理的文件数量
- 保留列表
- 待处理列表

每条预览记录至少应包含：

- 文件名
- 完整路径
- 匹配状态
- 计划动作
- 备注

### 7.7 执行要求

用户确认后，系统按所选策略处理原片目录中未匹配文件。

执行阶段要求：

- 支持进度展示
- 支持结果汇总
- 支持失败项列表
- 不因单个文件失败而中止整个任务

### 7.8 安全要求

该功能属于高风险删除类操作，必须满足：

1. 未生成预览时不可执行
2. 执行前必须二次确认
3. 永久删除时需要额外强化提示
4. 结果页需明确展示已处理数量、失败数量和失败原因

### 7.9 异常与边界

需要覆盖以下情况：

- 成片目录为空
- 原片目录为空
- 成片目录不存在可识别文件
- 原片目录不存在可处理文件
- 文件名大小写差异带来的匹配问题
- 文件无权限删除
- 文件被占用导致删除失败
- 部分成功、部分失败

### 7.10 验收标准

满足以下条件时视为该功能通过验收：

1. 工具能正确识别成片文件名主体集合。
2. 工具能正确找出原片目录中所有同名主体文件并保留。
3. 预览结果中的保留列表和待处理列表正确。
4. 删除策略与用户选择一致。
5. 失败项有明确反馈，整体任务不中断。

## 8. GUI 与交互要求

### 8.1 主结构

主界面建议采用以下结构之一：

- 顶部页签切换两个功能
- 左侧导航 + 右侧工作区

V1 以清晰、稳定、低学习成本为优先。

### 8.2 通用交互约束

两个功能都必须遵守以下流程：

1. 选择输入
2. 配置规则
3. 生成预览
4. 用户确认
5. 执行任务
6. 展示结果

### 8.3 状态反馈

系统需要提供以下反馈能力：

- 加载中 / 扫描中状态
- 任务执行进度
- 成功 / 失败 / 跳过的统计
- 错误提示
- 任务完成摘要

### 8.4 禁止误操作

- 在关键参数不完整时禁用“预览”或“执行”
- 在未完成预览时禁用“执行”
- 危险操作必须弹出确认对话框

## 9. 数据结构与实现接口约束

本节用于为后续实现提供稳定的任务模型，减少开发阶段再做产品决策。

### 9.1 时间修改任务

输入至少包括：

- 目标文件集合
- 时间偏移量
- 待修改时间字段集合

输出至少包括：

- 逐文件预览结果
- 逐文件执行结果
- 汇总统计
- 失败原因

### 9.2 原片筛选任务

输入至少包括：

- 成片目录路径
- 原片目录路径
- 匹配规则
- 删除策略

输出至少包括：

- 保留列表
- 待处理列表
- 逐文件执行结果
- 汇总统计
- 失败原因

### 9.3 预览记录

每条预览记录至少应包含：

- 文件路径
- 文件名
- 原始值
- 目标值
- 状态
- 备注或错误信息

## 10. 平台兼容与风险说明

### 10.1 平台目标

V1 目标平台为 macOS 与 Windows。

### 10.2 已知差异风险

实现阶段需重点关注以下差异：

- macOS 与 Windows 对“文件创建时间”的读写方式不同
- 回收站实现机制不同
- 文件系统权限模型不同
- 文件锁行为不同

需求层面要求：

- 用户可感知失败原因
- 平台差异不应导致静默失败
- 设计与实现文档中需要明确平台特定处理方案

## 11. 非功能要求

### 11.1 性能

- 面向个人日常使用场景
- 处理大量文件时界面不能完全失去响应
- 扫描、预览、执行过程需有进度反馈

### 11.2 稳定性

- 任何单文件异常都不应导致应用崩溃
- 批处理任务应尽量保证可部分成功

### 11.3 可用性

- 主要操作流程应在 3 到 5 个步骤内完成
- 默认文案应明确说明风险与结果

### 11.4 运行环境约束

- Python 必须在项目 `.venv` 虚拟环境中运行。
- 开发、测试、启动命令默认基于项目本地虚拟环境执行。
- 项目文档、脚本和后续自动化流程不得默认依赖系统全局 Python。
- 若检测到未进入虚拟环境，后续实现应提供明确提示。

## 12. 测试场景

### 12.1 时间修改功能测试

1. 单文件偏移预览正确
2. 多文件批量偏移预览正确
3. 仅修改创建时间时，其他字段不变
4. 仅修改修改时间时，其他字段不变
5. 仅修改拍摄时间时，其他字段不变
6. 无 EXIF 文件可正常提示
7. 不支持写 EXIF 的文件格式有明确提示
8. 只读文件失败时有明确反馈
9. 被占用文件失败时有明确反馈
10. 部分成功场景统计正确

### 12.2 原片保留功能测试

1. 成片 `IMG_0001.jpg` 能保留原片目录中同 basename 的不同扩展名文件
2. 无匹配文件时预览结果正确
3. 全部匹配时预览结果正确
4. 部分匹配时保留和待处理列表正确
5. 回收站模式执行正确
6. 永久删除模式执行正确
7. 删除失败时结果中能看到失败原因
8. 空目录输入时有明确提示

### 12.3 GUI 流程测试

1. 参数不完整时不能执行
2. 未预览时不能执行
3. 删除类操作必须二次确认
4. 长任务过程中有进度反馈
5. 完成后有汇总结果页或结果区域

## 13. 默认实现建议

以下内容属于默认建议，不等同于强制需求，但可作为后续开发基线：

- Python 3.12
- 使用项目根目录下的 `.venv` 作为默认虚拟环境
- PyQt6
- 文件扫描、预览计算与执行分层设计
- 预览与执行共用同一套任务模型，确保结果一致

## 14. 结论

V1 的产品边界已经明确：

- 聚焦两个高频场景
- 强调本地处理
- 强调预览优先
- 强调危险操作确认
- 强调双平台可用

后续进入设计与开发阶段时，应在不偏离本需求文档的前提下，补充具体模块设计、任务执行模型、异常处理方案和测试实现方案。
